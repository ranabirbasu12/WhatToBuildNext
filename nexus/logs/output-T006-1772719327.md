**v0.2 Architecture Recommendation**

1. **Dispatch: move to MCP, but keep shell as fallback in v0.2**
- Yes, switch primary transport to `codex mcp-server` (stdio MCP), because your current `codex exec` pipe + timeout flow is brittle and hard to evolve.
- Keep `nexus-dispatch.sh` as a compatibility path behind a config flag until MCP is proven in production-like usage.
- Practical rollout:
1. Add `dispatchBackend: "shell" | "mcp"` in config.
2. Build `nexus-mcp-dispatch.sh` (or equivalent wrapper) with the same inputs/outputs as current dispatch.
3. Keep current logging (`dispatches.jsonl`, `usage.json`) unchanged so metrics remain comparable.
4. Run both backends for 1 week on mirrored non-destructive tasks, compare failure rate/latency/output quality.
5. Cut over default to MCP; retain shell as break-glass fallback.

**Trade-offs**
- MCP gains: structured protocol, fewer quoting/stdio hacks, cleaner tool contracts, easier extension (board/knowledge as tools), better long-term maintainability.
- MCP costs: more moving parts (server lifecycle, retries, health checks), harder initial debugging, version compatibility management.
- Net: worth it for v0.2, but only with dual-path fallback during transition.

2. **Self-improvement loop design**
- Build a mandatory post-task “Reflect” stage after every dispatch.

**(a) Automatic knowledge extraction**
1. After task completion, run a reflection prompt that emits strict JSON (`fact`, `recommendation`, `type`, `confidence`, `evidence`, `tags`, `files`).
2. Only persist entries passing rules:
- has concrete evidence (`file/line`, error text, or test result)
- non-duplicate (semantic similarity check)
- confidence threshold (`medium+`)
3. Add `sourceTaskId`, `outcome` (`success|failed|retry`), and `expiresAt` for hygiene.

**(b) Detect recurring failures and adapt**
1. Add failure taxonomy in dispatch logs (`timeout`, `bad_spec`, `env_missing`, `test_flake`, `security_review_fail`, etc.).
2. Maintain rolling counters per taxonomy + per task type.
3. Trigger adaptation rules, e.g.:
- `bad_spec` spikes -> enforce acceptance-criteria template before dispatch.
- `timeout` spikes -> auto-split tasks and reduce context payload.
- repeated retry on same file -> force reviewer mode + human checkpoint.

**(c) Improve prompts/context over time**
1. Store prompt template version per dispatch.
2. Track outcome metrics by template (`pass rate`, `retries`, `review defects`).
3. Use lightweight A/B on prompt variants for similar task classes.
4. Context builder should rank knowledge by `relevance * recency * success_impact`, not just tag match.
5. Auto-prune stale/low-value knowledge entries.

3. **Other high-impact upgrades**
1. Replace JSON/JSONL state with SQLite (single file) for atomic writes, locks, better IDs, and queryable analytics.
2. Add an event log (`task_created`, `dispatched`, `reviewed`, `retried`, `escalated`) and derive board state from events.
3. Add policy gates pre-dispatch:
- dependency check
- required acceptance criteria
- max context size budget
- mode/assignee validity
4. Add outcome quality score per task (spec pass, tests pass, review severity) to drive routing decisions automatically.
5. Add nightly “system reflection” job: summarize top 3 failure modes + top 3 winning patterns and update project state/knowledge suggestions.
